OpenMPT's Design Problems: and my solutions.

Hi, i’m bored again. This article is mostly just me taking a break from other projects. One of these projects is my upcoming album, which takes a lot of time to make due to it’s complexity, my lack of ideas etc. But also one of the reasons is OpenMPT’s general crappiness in terms of both UI and UX. The entire team while developing ModPlug Tracker made some really questionable decisions, and i honestly want to correct them. Maybe not as a fork, not as a proof-of-concept hack, but as an idea with some mockups. Ok, let’s get into it.

How OpenMPT works, and how it looks while doing it

openmpt running on windows 7, taken from wikipedia

Let’s first look at OpenMPT’s design critically. The first thing i should note - mostly because it’s not apparent when looking at the picture - is that ModPlug Tracker is a multi-window application, similar to old versions of the Windows explorer, before it got reworked in Windows 95. It is mostly a good thing though, it means you can easily work on many modules in a single instance of the application. It’s also very well done, letting the user even minimize the module window to the bottom of the screen. The only problem i have with this feature is the fact that the window control buttons are slightly misplaced and hard to see while maximized. Now we should look at the 2 bars on the top, one of which is the tabbed module window bar, and the one on the top of the window is the general application bar. I also want to later talk about the menu bar, but let’s leave it for later. The bars have some duplicate buttons which shouldn’t really be there in my opinion. Both stop buttons do the same thing, and inexperienced users might have a problem differentiating these 2 sections. Also the record button on the pattern bar could be moved to work globally, saving some screen real estate. The application bar is really sparse, having many unnecessary buttons while not having enough options to be really useful. Things like the tempo and “speed” input fields shouldn’t be global, since they differ with every module, and are also available in the “general” tab of the module window. Much of the info on the bottom status bar could also be moved to the top. The module window bar is resizable, and it shouldn’t. i would make it a fixed size, especially with the widgets we see here. The file browser is really flawed, based on a 2-window system that is confusing. The top view only views your midi library and your current modules, and that works fine. But the bottom one is practically useless, mostly because it fails at being a sample browser. It doesn’t show directories in a tree - or in any way for that matter - , and when changing the current directory to look at some samples you’re greeted with the default windows file picker, which just makes everything clunky. This UI element is only marginally better than just opening every sample individually and playing it. My solution would be to move everything to the top panel, and remove the bottom one completely. The browser could be also used to show things like VST plugins and such, making it more universal in the process. Ok, so i said that i was going to talk about the default windows menu bar on the very top. It’s really weird since many of it’s functions, that are in the same place in many other applications (undo, redo in “Edit” and stuff like that), have other buttons in different places, stuff like that really needs a cleanup, and a lot of the visual clutter could be removed by just moving advanced features in the top menu, which exists here for a reason.

My “Solution”

my mockup containing some simple changes

First thing: remove visual clutter. The first thing i did was removing some unneeded buttons that don’t do anything major. All of that should still be accessible in the menu bar anyways. Second, i have decided to remove the bottom browser view, merging everything together. I also have removed the status bar, saving some creen real estate. The unintuitive placement of the play and record buttons has been corrected, while also moving the tempo boxes to the module window, mostly because the tempo changed with every module. All of these changes should be really easy to implement generally, and it shouldn’t take much work. Probably, i’m not an expert on the win32 api. You can obviously see some more changes here, but i don’t feel like turning this int o a bucket list of everything i have changed.

I have also made the mixer more approachable, by removing the flawed tabbed design and making it scrollable like in most software.

Conclusion

OpenMPT can be vastly improved with some simple changes. In a way i’m really frustrated with the development team that doesn’t make any changes, and just maintains the program. I personally now consider OpenMPT abandonware, even if it gets updates every other month. I’m getting tired, i’ll upload this tomorrow. Have fun out there!
